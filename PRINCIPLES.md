# Plugin + Skill 配合原则

## 核心定义

- **Plugin** = 确定性操作。代码强制执行，不需要也不允许模型决策。
- **Skill** = 开放性决策。提供方法论和参考，模型自主判断怎么做。

两者**不是先后关系，而是穿插配合**。任务执行过程中，随时可能需要 Plugin（遇到确定性操作）或 Skill（遇到需要判断的操作）。

## 什么时候用 Plugin

**答案只有一个的时候。**

- "这台机器是 macOS 还是 Linux？" → 只有一个答案 → Plugin 检测
- "保存快捷键是 Cmd+S 还是 Ctrl+S？" → 平台决定 → Plugin 给出
- "xdotool 装了没有？" → 事实 → Plugin 检查

特征：**不需要模型想，想了也是浪费。**

## 什么时候用 Skill

**做法有多种、需要判断的时候。**

- "这个按钮在屏幕哪个位置？" → 需要看截图 → Skill 流程（OCR→检测）
- "操作失败了怎么恢复？" → 需要分析原因 → Skill 给方法论，模型决策
- "先填名字还是先填地址？" → 策略选择 → 模型自己判断

特征：**需要模型的理解力和判断力。**

## 穿插模式

实际任务执行中，两者交替出现：

```
Skill: 模型看截图 → 决定要点击"Save"按钮
Plugin: 检测到 Linux → 用 pyautogui.click(x, y)（不是 pynput）
Skill: 模型看到弹窗 → 判断应该点"Keep Current Format"
Plugin: 用 xdotool type 输入文本（不是 typewrite，因为有特殊字符）
Skill: 模型验证结果 → 决定下一步操作
```

不是"先 Plugin 后 Skill"，而是**该确定时确定，该判断时判断**。

## 密度变化

唯一的"阶段性"特征：**任务开始时 Plugin 密度高，后期密度低。**

- **开始**：模型不了解环境 → 需要大量确定性信息（平台、工具、快捷键、分辨率）→ Plugin 多
- **中期**：模型已经知道环境 → 主要在做具体操作 → Skill 多，Plugin 偶尔（遇到新的确定性问题时）
- **后期**：模型对这个任务类型已经有经验（memory）→ Plugin 和 Skill 都少 → 模型自由发挥

这不是"阶段切换"，而是**密度的自然变化**。就像学开车：
- 刚上路：教练说的多（Plugin），你自己判断的少
- 熟练后：教练偶尔提醒（Plugin），大部分你自己开（Skill）
- 老司机：教练不用说（Plugin 几乎没有），你完全自主（Skill 也不怎么需要）

## 设计指南

### 设计 Plugin 时

1. **只做确定性的事** — 如果模型有可能需要做不同选择，那就不该是 Plugin
2. **注入要精简** — 只给模型需要的最小信息，不要一股脑全塞
3. **随时可调用** — Plugin 不只在开头运行，任务中途遇到确定性问题也可以调用
4. **不要替代模型思考** — Plugin 告诉模型"你在 Linux 上"，但不告诉模型"你应该点这里"

### 设计 Skill 时

1. **提供方法论而非指令** — "截图→OCR→找坐标→点击"是方法论，"点击 (500, 300)"是指令
2. **留出决策空间** — 模型可能发现更好的做法，不要限死
3. **可以触发 Plugin** — Skill 流程中如果遇到确定性操作，可以调用 Plugin 提供的能力
4. **积累记忆** — 模型学到的东西应该减少未来对 Skill 指导的依赖

### Plugin 和 Skill 的交互

```
Plugin 可以为 Skill 提供：
  • 环境信息（平台、工具、配置）
  • 确定性工具（检测到的最佳输入方式）

Skill 可以触发 Plugin：
  • 遇到需要确定性操作的步骤（如：这个键盘布局下怎么打特殊字符？）
  • 需要环境重新检测（如：连接到新的远程机器）

两者共享：
  • Memory（Plugin 检测的环境信息可以存到 memory，Skill 操作经验也存到 memory）
  • Context（Plugin 注入的信息 Skill 可以直接使用）
```
